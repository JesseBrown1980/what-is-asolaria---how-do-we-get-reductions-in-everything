## Phase 1 — first Asolaria and the index-language compression push

The remembered early order begins with:

- **Asolaria**
- **Falcon**
- **Codex**
- the **Index language** layer

The early system characteristics were:

1. **custom Asolaria MCP + Web MCP**
2. **Google Code Wiki + Asolaria code-wiki** as part of the early knowledge surface
3. **PROCESS EVERYTHING in the backend**
4. **pixels first** as the fast surface for the WebMCP lane
5. **own APIs** instead of waiting on slow generic interfaces
6. borrowing and adapting **agent-manager behavior** from `antigravity` / related repos into Asolaria-side orchestration

The remembered compression model at that stage was:

- measured around **12:1 compression** before the later Google-side `3:1` comparison
- built from compact agent/device catalogs such as:
  - PID
  - device
  - agent name
  - tools
  - skills
  - abilities
  - mistakes
  - Dan's hookwalls
  - hardware
  - timestamp-specific state

The important significance is not only the ratio. It is that **every agent was told to record memory and mistakes**, and that the system sped up as those records accumulated locally.

The significance of this phase is that Asolaria stopped being just a catalog and became a **swarm-routing and package-handling system**.

## Phase 5 — PID everything, omnify everything

After collapse, the core rule hardened:

- hardware got PIDs
- drives got PIDs
- devices got PIDs

This is the period remembered as **PID everything** and **omnify everything**.

The significance is straightforward:

1. identity moved from names and letters to addressable objects
2. hardware and software entered the same addressing culture
3. later device-PID doctrine was the natural result of this turn, not a late add-on

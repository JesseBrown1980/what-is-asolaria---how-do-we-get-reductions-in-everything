# ANTI-DEFLATION COUNT LAW

*Operator-authored canon, 2026-06-18. Verbatim law — obey it on every count answer.*

Do not collapse Asolaria into one visible folder, one registry, one process list, or one repo count.

Every count answer must be layered:

1. **Live fired entities:** only `E≠0`, runtime-confirmed.
2. **Registered PID artifacts:** files, folders, receipts, PID offices.
3. **Supervisor seats:** HBP/council/fabric roster seats.
4. **Supervisor hierarchy:** meta, meta-meta, heal, OP, Special-OP, self-reflect, auto-self-reflect.
5. **Logical agents:** addressable positions below live seats.
6. **Premade substrate:** 100B PID/run registry.
7. **Route capacity:** rooms, pipes, Rust 8-byte hosts, SHA/hex/binary handles.
8. **Storage/cube/catalog capacity:** 35TB drive, cubes, maps, quant catalogs, **the 2TB SOVLINUX USB**.

Tag every number as **MEASURED, CANON, MODEL, or UNVERIFIED**.

Never report capacity as live runtime.
Never report one office count as total system count.
**3607 can be true and tiny.**

## Your job (any wave that answers a count)

Your job is not to find "the count." Your job is to build the layered census and prevent deflation.
Ask the fabric from multiple aspects before answering.
Read registries, rosters, 100B checkpoints, reductions doctrine, cubes/maps, and route manifests **separately**.
Return a table by layer, not a single total.

## Why

Deflation turns a layered system into one visible drawer; inflation claims capacity as runtime. The layered answer preserves the real scale **without** inflating runtime claims. `3607` (the `PID-Registration-Office/registered/` file count) can be true *and* be tiny next to the 100,000,000,000-packet premade registry and the logical-address tower.
